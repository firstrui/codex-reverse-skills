## 验证码一律先拆两条链

从阿里、快手、百度这几类文章可以抽出统一套路：

- 一条是图像 / 位置 / 角度链
- 一条是轨迹 / 指纹 / 加密 / 验签链

例如：

- 百度旋转里 `fuid` 更像指纹字段
- `fs` 更像轨迹、点击、角度等数据的加密承载
- 阿里 `_rand` 则把值藏进 CSS 动画和样式终态

因此，验证码不要一上来就把所有字段混着看。

## 先找 style / log / verify 三段

验证码场景里，优先把请求分成：

- 初始化 / `style`
- 行为 / `log`
- 提交 / `verify`

这能帮助你快速判断：

- 哪些字段来自初始化返回
- 哪些字段来自用户轨迹
- 哪些字段只是最终封装

## 参数链与协议链的做法

### 小红书 `protobuf`

高价值经验：

- 看到 `jspb`、`array`、`readString/readBytes/readEnum` 这类特征，优先怀疑官方 `protobuf-javascript`
- 先找 `json -> builder/array -> proto -> b64/加密` 边界
- 已经能看出 message 层级和字段类型时，优先重写 `.proto`，再编译对应 JS

这类题不一定需要“无 schema 盲猜”，很多时候只是前端对象到 protobuf 的重建题。

### `rid` / `a_bogus` / `x-s3-s4e` / `desc`

共同经验：

- 先确认统一写入边界，不要只盯单个字段名
- 版本漂移时，先拿旧版可用脚本或旧版样本做差分
- 能插桩就先插桩，日志足够时不一定要完整反编译

特别是 `a_bogus`：

- 先比 `salt`
- 再比短数组 / 长数组转换
- 再比魔改 RC4 / 哈希内核

### 小程序网关 / 特殊协议

`rid` 一文的启发是：

- 抓包路径受限时，优先从代码桥接层或官方网关能力入手
- 找不到入参时，就从底层交互和调用桥往上反推

## 移动端网络栈 / QUIC / Hook

`TikTok` 与 `lazada` 两篇提醒：

- 有些题的第一阻塞不是参数，而是抓不到包
- 遇到 `QUIC`、So 层、JNI、代理失效，不要硬用 Web 逆向套路

优先顺序：

1. 先看日志
2. 再看网络栈选择与降级点
3. 再看 So / JNI 中真正消费参数的位置

如果只是为了定位关键参数，Hook 关键方法往往比全量静态分析更快。

## `内网穿透` 的定位

这篇是侧篇。

它更适合作为：

- 本地调试基础设施
- 远端环境联通
- 外部联机验证

不要把它误当成参数链或壳层恢复方法。

## 三个优先 probe

- 把验证码字段拆成“位置 / 轨迹 / 指纹 / 加密载荷”
- 找 `json -> builder -> encode` 或 `style/log/verify` 的统一边界
- 移动端先看日志和网络栈，再看 So / JNI / Hook 点

## 推荐 handoff

- 验证码参数双链：`$captcha-parameter-chain-bisection`
- 协议与参数回放：`$js-reverse-sign-replay`
- 无 schema 的 protobuf：`$protobuf-schema-backfill`
- QUIC 降级与 Android 抓包：`$android-quic-downgrade-hook-capture`
